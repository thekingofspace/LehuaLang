Messenger

Every Lehua program has a global called messenger: a broadcast system that reaches across threads. Publish a topic with some values, and every subscriber to that topic runs its callback, wherever it lives. No include header needed - it is always there, in the main thread and in every parallel worker.

Publish and subscribe

A Port is a pipe between exactly two threads, where one side pushes and the other side pops. The messenger is different: there is no queue to drain and no pairing. Publishing is fire and forget - whoever is subscribed at that moment gets a copy, and nobody subscribed means the message simply evaporates.

MethodReturnsDescription
messenger:Publish(topic, ...)numberSends the values to every subscriber of topic in the whole program, including this thread. Returns how many subscribers were reached.
messenger:Subscribe(topic, callback)SubscriptionRuns callback(...) for every publish to topic, from any thread.
messenger:Once(topic, callback)SubscriptionLike Subscribe, but the subscription removes itself after the first delivery.
messenger:Subscribers(topic)numberHow many live subscriptions the topic has across all threads.

A Subscription has Unsubscribe(), Topic(), and IsActive().

Callbacks run at home

When an event arrives, the callback runs in a fresh coroutine on the thread that subscribed, never on the publisher's thread. It can yield, use task.wait, do network calls, or publish again. Values travel like Port values: nil, booleans, numbers, strings, and plain tables, copied not shared. A mem cache handle also crosses, which is the door to sharing bigger state.

src/main.luau
parallel("@self/shard.luau", 1)
parallel("@self/shard.luau", 2)

local done = 0
local sub
sub = messenger:Subscribe("shard-ready", function(id)
    done += 1
    print("shard " .. id .. " is ready")
    if done == 2 then
        messenger:Publish("announce", "all shards up")
        sub:Unsubscribe()
    end
end)

messenger:Once("announce", function(text)
    print("shard " .. id .. " heard: " .. text)

Lifetime

A live subscription keeps its thread alive, the same way a bound heartbeat does: the program will not exit while something is still listening. Call Unsubscribe() when a subscription's work is done. Errors inside a callback are printed and do not affect the publisher or other subscribers.

Serial too

Subscribing is not just for workers. The main thread can subscribe to its own topics, and a publish from the main thread reaches its own subscriptions as well - the callback still runs in its own coroutine, shortly after the publish returns.
