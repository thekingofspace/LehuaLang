Included files

The --#includestring header bakes a file into your program, and the frominclude keyword reads it back as a string at run time. This is how you ship data next to your code: a config file, an HTML template, a shader, an embedded image, a license. When you build a program, the file's bytes are packed into the executable, so the running program carries its data with it and does not need the original file on disk.

Included files versus fs

Use fs when you want to read or write files that live alongside the program at run time. Use an included file when you want the data to travel inside the program itself. An included file is read only and always there, even when the executable is copied somewhere new.

Declaring and reading

Add a header near the top of a file. Each entry gives a name and a path. The name is how you reach the file from code; the path says which file to bake.

src/main.luau
--#includestring[Config: "./config.json", Banner: "./banner.txt"]

local config = frominclude("Config")
local banner = frominclude("Banner")

print(banner)
local settings = serde.decode("json", config)

frominclude(name) returns the file's contents as a string. The string holds the exact bytes of the file, so binary data such as an image or a compressed archive comes back untouched. Reading the same name again returns the same value, so it is cheap to call wherever you need it.

Paths are always file relative

An included path is resolved against the folder of the file that declares it, never against roots or aliases. This is the one rule, and it never changes:

In the headerResolves to
./data.txtA file next to the current file.
@self/data.txtThe same thing. @self is the current file's folder.
../shared/data.txtUp one folder, then into shared.
assets/logo.pngA plain path is relative to the current folder too.

Because every path is anchored to the declaring file, moving a folder keeps its included files working without changing the header.

Included files are file specific

An included file belongs to the one file that declared it. Only that file can read it, and only by the name it chose. A module that wants its own data declares its own header, even for the very same file on disk.

src/report/page.luau
--#includestring[Template: "@self/template.html"]

local html = frominclude("Template")   -- this file's own include
-- frominclude("Config") here would error: Config was declared in main.luau, not here

Naming

The name is a label of your choosing, separate from the path. If you leave the name out, it defaults to the path itself, which is handy for a one off.

--#includestring["./notes.md"]

local notes = frominclude("./notes.md")

Declaring the same name twice keeps the first path. Spread entries across several --#includestring lines or list them in one, whichever reads better.

What happens at build time

When you run lehua build, each included file is read from disk and packed into the executable. During lehua run the file is read the same way, so both behave alike. If a declared file cannot be found while building, the build prints a note and continues; the program will still run, and the missing file only becomes an error if frominclude actually reaches for it and cannot find it on disk either.

Keep large data in mind

Included files live inside the executable and in memory once read. That is perfect for configs, templates, and modest assets. For very large or optional data that the program may never touch, a file read through fs at run time is the lighter choice.
