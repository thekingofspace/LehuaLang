Native DLLs

Lehua calls native code through small DLLs (shared objects on Linux and macOS). There are two ways to reach one: an inject header that binds a DLL as a global for a single file, or the dll library that opens a DLL as a value you can pass around and re-export. Both load the same kind of DLL and both get packed into your executable when you build.

The inject header

An inject header loads a DLL and binds it as a global for that one file only. The DLL is found next to the script that injects it.

src/main.luau
--#inject[mathext.dll]

print(mathext.double_it(21)) -- 42

The global is scoped to the file that injects it, so a module that injects a DLL keeps that global to itself. That is usually what you want: the module wraps the native calls and returns a normal table, and the rest of your program uses that table without ever seeing the DLL. Because the returned functions close over the injected global, calling module.something() from a file that never injected the DLL works fine.

src/mathwrap.luau
--#inject[mathext.dll]

-- mathext is visible here, in this module only.
return {
    double = function(n) return mathext.double_it(n) end,
}
src/main.luau
local math = require("./mathwrap")
print(math.double(21)) -- 42, no inject header needed here

The dll library

The dll library does the same thing at run time, as a value instead of a header. Include it, then open a DLL by a path relative to the current file. You get back a table whose fields are the DLL's exports, ready to call.

src/main.luau
--#include[dll]

local mathext = dll.open("./native/mathext.dll")
print(mathext.double_it(21)) -- 42
FunctionReturnsDescription
dll.open(path)tableOpens a DLL, resolved next to the current file, and returns its exports as a table. dll.load is the same function.

This is the tidy way to keep native code behind a normal Luau module: a module opens the DLL and returns a wrapper table, and every other file just requires that module. The library resolves paths the same way the inject header does, relative to the file that calls dll.open, so the DLL is embedded into your executable at build time just like an injected one.

How calls run

Calls into a DLL run on a background thread, so a slow call blocks only the coroutine that made it. When you build the program, the DLLs you reference (through either an inject header or dll.open) get packed into the executable. A DLL that is missing at build time is a warning, not an error, and the built program will look for it next to itself at run time. On Linux the same mechanisms load a shared object, for example --#inject[libmathext.so] or dll.open("./libmathext.so").

The DLL interface

A Lehua ready DLL exports one C function per call, plus a lehua_free function. Each call takes JSON bytes for the arguments and returns JSON bytes for the result.

plugin.c
// input:  a JSON array of the call arguments, for example [21]
// output: a JSON object {"ok": value} or {"err": "message"}
uint8_t* double_it(const uint8_t* in, size_t in_len, size_t* out_len);
void     lehua_free(uint8_t* ptr, size_t len);
Lehua is built in Rust on mlua and tokio. Made for people who love Luau.
