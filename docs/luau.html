Library Reference

The luau library

The luau library works with Luau code as data. It can turn a string of source code into a function you can call, run a string right away, and compile source into bytecode. This is the tool for plugins, config scripts, and REPL-style features.

Including luau

--#include[luau]
Only run code you trust

A string that you load and call can do anything the rest of your program can do. Never feed user input, network data, or file contents you do not control into luau.load or luau.eval. If you must run outside code, give it a small environment table so it can only reach what you put there.

Functions

FunctionReturnsDescription
luau.load(source, options?)functionCompiles a string into a function you can call later.

Loading code

luau.load(source, options?) compiles the string and returns a function. Nothing runs until you call that function. Compilation happens immediately, so a syntax error raises a Lua error from load itself; wrap the call in pcall if the source might be malformed.

Arguments you pass when calling the returned function arrive inside the loaded code as ..., the vararg expression. The loaded code can also return values, and you get them back from the call.

src/main.luau
--#include[luau]

local add = luau.load("local a, b = ...; return a + b")
print(add(2, 3)) -- 5

local ok, err = pcall(luau.load, "this is not luau")
if not ok then
    print("bad source:", err)
end

The optional options table has two fields:

OptionTypeDescription
options.namestring?The chunk name shown in error messages and stack traces. Defaults to "luau.load".
options.environmenttable?A table the loaded code uses instead of the normal globals. See below.

What loaded code can see

By default, loaded code sees the normal globals: print, string, table, math, pcall, and the rest of the standard library.

It does not see require, parallel, __dirname, __filename, or included libraries like fs. Lehua hands those to each module privately rather than making them globals, so a loaded chunk never inherits them. If loaded code needs one of them, pass it in yourself, either as a call argument or through the environment table.

Giving an environment table replaces the globals entirely. Every global lookup inside the loaded code goes to your table instead, including the standard library, so even print is gone unless you add it. This makes a simple sandbox: the code can only touch what you put in the table.

src/main.luau
--#include[luau]

local script = luau.load("report(counter + 1)", {
    name = "sandbox",
    environment = {
        counter = 41,
        report = print,
    },
})
script() -- 42

Evaluating code right away

luau.eval(source, options?) is load-and-call in one step. It compiles the string, runs it with no arguments, and returns everything the code returns, including multiple values. It takes the same options table as load. A syntax error or a runtime error inside the code raises a Lua error, so use pcall for input you are not sure about.

The source must be a full chunk, so write return in front of an expression. "2 + 3" on its own is a syntax error; "return 2 + 3" works. If you need to pass arguments, use luau.load and call the function yourself.

src/main.luau
--#include[luau]

print(luau.eval("return 2 + 3")) -- 5

local x, y = luau.eval("return 10, 20")
print(x, y) -- 10 20

Compiling to bytecode

luau.compile(source, options?) runs the Luau compiler and returns the bytecode as a string, without executing anything. Bytecode is binary data, not readable text, but a Luau string can hold it fine, so you can measure it, hash it, or save it with fs.write. A syntax error raises a Lua error.

luau.load and luau.eval take source text, and there is no function that runs a saved bytecode string, so treat compile as a tooling function: checking that source parses, comparing output sizes across settings, or feeding build pipelines.

The optional options table sets the compiler levels. Each level is a number from 0 to 2.

OptionDefaultDescription
options.optimizationLevel10 is no optimization. 1 applies optimizations that keep debugging intact. 2 adds aggressive ones, like inlining, that make stack traces harder to read.
options.debugLevel10 strips debug info. 1 keeps line numbers and function names for stack traces. 2 also keeps local and upvalue names.
options.coverageLevel00 records no coverage. 1 tracks which statements ran. 2 also tracks expressions.
src/main.luau
--#include[luau, fs]

local source = fs.read("./plugin.luau")

local ok, bytecode = pcall(luau.compile, source, {
    optimizationLevel = 2,
    debugLevel = 0,
})
if ok then
    print("compiles cleanly:", #bytecode, "bytes")
    fs.write("./plugin.bin", bytecode)
else
    print("syntax error:", bytecode)
end
